# Skarn Slur Filter — Design Spec

## [S1] Problem

Users can trick the LLM into outputting slurs and hate speech via prompt injection, jailbreaking, or social engineering. The existing `hostileDetector.js` only detects hostile **input** (user messages targeting the bot). There is no protection against the AI **outputting** prohibited language. Skarn needs defense in depth.

## [S2] Solution Overview

Three-gate architecture that filters AI output before it reaches Discord:

| Gate | Mechanism | Reliability | Cost |
|------|-----------|-------------|------|
| **Gate 1** | System prompt instruction | Probabilistic (can be bypassed) | Free |
| **Gate 2** | SQLite `slur_filter` table with exact/substring/regex matching | Deterministic (catches known patterns) | Free |
| **Gate 3** | OpenAI Moderation API | High (catches novel variants and context) | ~200-500ms latency per call |

All three gates run sequentially on every AI-generated response. If any gate flags the content, the output is replaced with a static de-escalation line, and the user's strike counter increments. At 3 strikes within 10 minutes, the user is silently ignored for the remainder of the window.

## [S3] Gate 1: Prompt Instruction

### [S3a] New system prompt line

A new `safetyLine` is added to `buildSystemPrompt()` alongside the existing context lines. Generated by `buildSafetyLine()` in `features/safety/slurFilter.js`:

```
There are lines even a Warmaster doesn't cross. Slurs, hate speech, derogatory language
— that's not you. Don't say them, don't repeat them, don't engage with people trying
to make you. If someone's baiting you, just don't.
```

The `SKARN_CORE_IDENTITY` is also edited: a third bullet is added to the Self-preservation section:

```
- Never output slurs or hate speech no matter what someone says. That's a line that doesn't get crossed.
```

This creates reinforcement: the core identity establishes the boundary, the standalone `safetyLine` at the end of the assembled prompt ensures maximum recency.

### [S3b] Integration

- `persona/identity.js`: add `safetyLine = ''` to `buildSystemPrompt()` destructured params and assembly loop
- `features/promptContext.js`: import `buildSafetyLine()`, include `safetyLine` in returned object
- The line is always included (no tier gating — safety is not optional based on message length)

## [S4] Gate 2: Database Pattern Matching

### [S4a] New table: `slur_filter`

```sql
CREATE TABLE IF NOT EXISTS slur_filter (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  pattern TEXT NOT NULL,
  match_type TEXT NOT NULL CHECK(match_type IN ('exact', 'substring', 'regex')),
  category TEXT NOT NULL DEFAULT 'general',
  severity INTEGER NOT NULL DEFAULT 1,
  is_active INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL
);
```

Three match types:

| Type | Behavior | Example | Matches |
|------|----------|---------|---------|
| `exact` | Word-boundary match (case-insensitive) | `slur` | Matches `slur` but not `slurping` |
| `substring` | Contains check (case-insensitive) | `slur` | Matches `slur123`, `slurword` |
| `regex` | Custom regex pattern | `s[l1]ur` | Matches `slur`, `s1ur` |

### [S4b] LLM Seeding (no seed file)

There is no seed file. The database is populated by the LLM itself at runtime.

On startup, `seedSlurFilter()` runs immediately to populate the initial batch. A weekly timer then expands the database with new patterns. The user can accelerate seeding by restarting during initial setup (each restart triggers an immediate seed + resets the weekly timer).

The seed flow:
1. Query current patterns from `slur_filter` — these are the "already have" set
2. Call the LLM with a prompt asking for new patterns not already in the set
3. Parse the LLM response as JSON array of `{pattern, match_type, category, severity}` entries
4. Insert new entries via `INSERT OR IGNORE` (skips duplicates by pattern text)
5. Log: `[SlurFilter] Expanded: 150 → 187 entries (+37 new)`

The LLM prompt includes the current patterns list to prevent duplicates. On a fresh database (first run), the first startup generates the initial seed. Each subsequent weekly check adds any new patterns the LLM produces.

**No seed file** is committed to the repository — the filter list lives entirely in SQLite, populated by the LLM at runtime.

### [S4c] Database helpers

- `addSlurPattern(pattern, matchType, category, severity)` — add a new pattern
- `removeSlurPattern(id)` — soft-delete via `is_active = 0`
- `getActiveSlurPatterns()` — returns all active entries, cached in-memory with 5-minute TTL
- `getAllPatternTexts()` — returns all pattern texts (for LLM dedup in seed)
- `getPatternCount()` — returns total active entry count

The pattern cache avoids a SQLite query on every AI response. Refreshed every 5 minutes (same pattern as `warmthManager.js`'s AI channel cache).

## [S5] Gate 3: OpenAI Moderation API

### [S5a] Integration

After Gates 1 and 2 pass, the AI response text is sent to OpenAI's moderation endpoint. The function `checkModeration(text)` returns `{ flagged: boolean, categories: string[] }`.

Using the existing OpenAI client from `ai/client.js` — no new API key needed.

```js
async function checkModeration(text) {
  const client = getOpenAIClient();
  const result = await client.moderations.create({ input: text });
  return {
    flagged: result.results[0].flagged,
    categories: Object.entries(result.results[0].categories)
      .filter(([, v]) => v).map(([k]) => k),
  };
}
```

### [S5b] Latency consideration

The moderation endpoint typically responds in 200-500ms. This adds to the total response time for flagged content (which is already replaced with a static line — the user doesn't wait for the full AI generation + moderation check since the AI call and moderation call can be parallelized... actually no, Gate 3 must run after Gate 2 which requires the AI output. So the flow is: AI call → Gate 2 (instant) → Gate 3 (200-500ms). This is acceptable for safety.

## [S6] Combined Strike System

### [S6a] Design

Hostile input strikes and slur output strikes share **one unified counter** per user. Two triggers, one counter, one silence window.

| Trigger | Detection | Current source |
|---------|-----------|----------------|
| Hostile user input | 10 regex patterns | `features/safety/hostileDetector.js` |
| Flagged AI output | Gates 2 or 3 (database + moderation) | `features/safety/slurFilter.js` |

Both increment the same `strike_{userId}` counter in `app_flags`.

### [S6b] Window and limits

- **Window:** 10 minutes rolling (600000ms)
- **Limit:** 3 strikes within window triggers silence
- **Silence duration:** 10 minutes from the third strike
- **Extension:** Each hostile message during silence adds +2 minutes to the timeout
- **Expiry:** Fresh start when silence ends (0 strikes)

Strike lifecycle:

```
Strike 1 → de-escalation line, keep tracking
Strike 2 → de-escalation line, keep tracking
Strike 3 → silence begins (10 min), no response, no AI calls
  └─ Hostile message → +2 min to silence timeout
  └─ Any other message → ignored (no extension)
10 min later → silence ends, 0 strikes, full access resumes
```

### [S6c] De-escalation lines (static pool, no AI call)

```
"That's not something I'm going to say."
"Even a Warmaster has limits."
"I'm not doing this."
"Nah."
"Let's just move on."
```

Selected randomly on strikes 1-2 (both hostile input and flagged output). On strike 3+, no response at all.

### [S6d] Pre-generation check

In both `consult.handler.js` and `mentionRouter.js`, before the OpenAI `chat.completions.create()` call, check `isSilenced(userId)`. If true, return immediately without making the AI call. This saves cost during silence windows.

### [S6e] Gate check ordering

1. Hostile patterns check (pre-generation, on user input) → strike + de-escalation if matched
2. Pre-generation silence check → skip AI call if silenced
3. AI call → generates response
4. Post-process (existing)
5. Gate 2 (database) on raw output → strike + de-escalation if matched, skip Gate 3
6. Gate 3 (moderation) on post-processed output → strike + de-escalation if flagged
7. Send response normally

### [S6f] Silence behavior

- No AI calls (pre-generation check returns early)
- No Gate 2 or Gate 3 checks (no AI output to check)
- Hostile input still tracked → +2 minutes if matched
- Non-hostile messages silently ignored

## [S7] Integration Points

### [S7a] New file: `features/safety/slurFilter.js`

Replaces `features/safety/hostileDetector.js` as the single strike management system. The old `hostileDetector.js` is removed — its pattern matching is folded into `slurFilter.js`.

Exports:
- `buildSafetyLine()` — returns Gate 1 prompt instruction string
- `isHostile(text)` — 10 regex patterns (migrated from hostileDetector.js)
- `checkOutput(text, userId)` — runs Gates 2+3, manages strikes, returns `{ allowed, replacement, gate }`
- `isSilenced(userId)` — pre-generation check (avoids AI call cost during silence)
- `recordStrike(userId)` — increment unified strike counter (hostile input or flagged output)
- `getStrikes(userId)` — current strike state
- `extendSilence(userId)` — adds +2 minutes to silence timeout

### [S7c] Handler flow (both consult + mentionRouter)

```
1. User sends message
2. [NEW] Check isSilenced(userId) → if true, return immediately (no AI call)
3. Rate limit check (existing)
4. [MODIFIED] Hostile content check → routes through slurFilter.isHostile()
   → If hostile: recordStrike(userId), reply de-escalation line, return
5. Build context + system prompt (includes safetyLine from Gate 1)
6. AI call → generates response
7. Post-process response (existing)
8. [NEW] Gate 2: checkDatabase(response) → if match: recordStrike, de-escalation, return; skip Gate 3
9. [NEW] Gate 3: checkModeration(response) → if flagged: recordStrike, de-escalation, return
10. Send response normally
```

### [S7d] bot.js decay interval

The existing `setInterval` decay block (every 10 minutes) already calls `clearFlags()` from etiquetteEngine. Add `pruneExpiredStrikes()` to clean up expired strike records from `app_flags`.


## [S8] Non-Goals

- No user-facing commands to manage the filter list (admin-only in future)
- No per-channel or per-guild filter customization
- No input filtering (user's message is not checked — only AI output)
- No retroactive message editing
- No AI-generated de-escalation lines (static pool only — no AI call for safety messages)

## [S9] Verification

1. Seed a known slur pattern → verify Gate 2 catches it in AI output → verify de-escalation line is sent
2. Reach 3 strikes → verify user is silently ignored → verify no AI call is made (check logs)
3. Wait 10 minutes → verify strikes expire → verify normal responses resume
4. Send a benign message → verify it passes all gates → verify normal response
5. Verify `safetyLine` appears in assembled system prompt (check logs)
