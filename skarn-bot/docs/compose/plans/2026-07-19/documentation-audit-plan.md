```markdown
# skarn-bot — Architecture & Design Context

## Purpose of this file
This file exists so architectural decisions are not re-litigated or
re-discovered from scratch. It captures why the bot is structured the way
it is — the constraints, trade-offs, and deliberate separations that
future work (by any model or human) should respect. Link this from README.

## Core architectural pattern
Vertical slice architecture: each feature owns its command definition,
handler, and data access together in `features/<feature>/`. No shared
"god" file per concern.

Persona identity (in `persona/identity.js`), role definitions (`persona/roles.js`),
and context assembly (`features/promptContext.js`) are the deliberately shared
layer — every AI-powered surface routes through the same
`buildSystemPrompt()` function, no exceptions. Even commands that need no
state/memory context pass empty strings rather than building their own
prompt inline.

The activation registry (`features/activation/activationRegistry.js`) is
a second shared cross-cutting concern: maps keyword phrases to commands
at startup, providing text-based alternatives to slash commands. Two routing
types: `'command'` (runs handler directly) and `'ai'` (injects a directive
and routes to the AI mention handler).

## Scoping conventions

| Subsystem | Scope | Key | Reason |
|-----------|-------|-----|--------|
| User memory / memory_entries | Per-guild | (user_id, guild_id) | Memories are contextual to a server |
| Channel state | Per-channel | channel_id | State tracks channel activity, not user |
| Relationship / Familiarity | Per-guild | (user_id, guild_id) | Familiarity is server-specific |
| Conversation graph | Per-guild-channel-user | (user_id, guild_id, channel_id) | DMs have no guild context (guild_id is irrelevant) |
| Guild config | Per-guild | (guild_id, key) | Server settings belong to server |
| Guild mood | Per-guild | guild_id | Mood is a server-level property |
| Server culture | Per-guild-channel-ngram | (guild_id, channel_id, ngram) | Language patterns are channel-specific |
| Realm (characters, inventory, etc.) | Per-guild | (user_id, guild_id) | Per-guild by design — Realm v1 had an inconsistent mix of scoping before being corrected |
| Confidant Mode | Account-wide | user_id | DMs have no guild context; this was an explicit design decision |
| Derived Memory (knowledge_graph) | Per-guild | (user_id, guild_id) | Matches the pattern set by user_memory/memory_entries |
| User preferences | Per-guild | (user_id, guild_id) | Preferences can differ per server |
| Attention state | Per-user-guild-channel | (user_id, guild_id, channel_id) | Triplescope because attention is context-dependent |
| Cooldowns | Varies | (user_id, channel_id) or channel_id | Depends on the cooldown type |
| Follow-ups | Per-guild | (user_id, guild_id, channel_id) | Tied to a specific conversation context |
| User profile | Per-guild | (user_id, guild_id) | Engagement and topics are per-community |
| Emotional context | Per-guild | (user_id, guild_id) | Emotional state is server-relative |

**Rule:** Any new AI-calling feature that stores state must declare its scoping
explicitly and justify why it doesn't follow the established per-guild
(user_id, guild_id) composite key pattern.

## Rate limiting and cost control

| Bucket | Window | Ceiling | Why separate |
|--------|--------|---------|-------------|
| General AI calls (rate_limits table) | 10 min sliding | Default 10/10min via lib/rateLimit.js | Protects API budget from burst usage |
| Per-user hourly cap (ai_usage) | 1 hour | 50/hr via lib/aiStats.js | Second layer — prevents single user from exhausting daily budget |
| Mention cooldown (mention_cooldowns) | 1s | 1 per user per channel | Prevents rapid @mentions from spamming AI |
| Interjection cooldown (interjection_cooldowns) | 5 min | 1 per channel | Random interjections are sparse by design |
| Active listen cooldown (active_listen_cooldowns) | 5 min | 1 per channel | Warmth manager active listening is sparse |
| Reaction cooldown (reactionSystem.js) | 60s | 1 per channel | Reactions are background flavor, not response |

Each major feature (general persona, Realm, Confidant, Derived Memory) has
its OWN bucket — reusing a shared bucket across features with different usage
shapes was tried and reverted once. A single Realm session or Confidant
conversation could exhaust a budget sized for occasional one-shot commands
within minutes. Any new AI-calling feature needs its own bucket unless
there's a specific documented reason to merge.

## Persona and role conventions

- `SKARN_CORE_IDENTITY` in `persona/identity.js` is the one unchanging voice
  constant, never duplicated or paraphrased elsewhere.
- Every AI command's role-specific instruction lives in `persona/roles.js`'s
  `roles` object — never inlined in a command file.
- Every role has a paired token budget in `roleTokenBudgets` — no command
  uses a bare/ungoverned max_tokens value. Current budgets range from 100
  (roast, compliment) to 1000 (realm).
- Every role has a `ROLE_NATURE` classification (`casual`/`moderate`/`serious`)
  in both `persona/roles.js` and `features/discordNative/postProcess.js` that
  controls post-processing behavior (lowercase injection, abbreviation addition,
  emoji scatter, message splitting).
- Temperature is set per role type: low (~0.2-0.3) for factual/extraction/
  summarization tasks, higher (~0.7-0.85) for narrative/conversational tasks.
- `ROLE_NATURE` is duplicated in both files (roles.js for context assembly,
  postProcess.js for response transformation) — this is established pattern,
  keep both in sync when adding roles.

**Note on token budget drift:** The discord-native-persona spec (2026-07-18)
proposed `consult: 900` to accommodate context awareness. The actual code
has `consult: 400`. The context injection was implemented but the token
budget was never increased — this is a known doc-code drift point.
```

```markdown
## Memory systems — what's separate and why

This project has FOUR (or more) distinct memory stores, deliberately kept apart:

### 1. user_memory (legacy — actively read) + memory_entries (unified — write target)
- **CRITICAL FRAGMENTATION:** These two tables are OUT OF SYNC.
- `user_memory`: Original table. STILL read by 19 command files (joke, roast, code,
  homework, song, story, etc.) via `getUserMemory()` in `db/database.js`. Nothing
  writes to it anymore except the legacy `addUserMemory()` function (dead export,
  not called anywhere).
- `memory_entries`: Newer unified table (`source` discriminator: `'etch'`,
  `'extracted'`, `'conversation'`). This is what `/etch` writes to
  (`etch.handler.js` → `addMemoryEntry()`). Also what `promptContext.js`
  reads for AI context assembly.
- `source='etch'`: User-stated facts (confidence 1.0, no decay in confidence per se).
- `source='extracted'`: Entities from knowledgeGraph.js (confidence starts
  at 0.5, decays over time).
- **Effect of fragmentation:** A user who runs `/etch` adds a fact to `memory_entries`,
  but the 19 creative commands (joke, roast, etc.) still read from `user_memory` and
  won't see it. Meanwhile `promptContext.js` reads from `memory_entries` for the
  AI context assembly, so the AI DOES see etch facts. The two tables have diverged.
- This is the single most important documentation finding — it's a real data bug,
  not just doc drift.

### 2. knowledge_graph
- Entity-extraction table (`entity_type`, `entity_name`, `context`, `confidence`).
- Written by `features/intelligence/knowledgeGraph.js` using AI analysis of
  conversations. Separate from memory_entries despite conceptual overlap.
- Per-guild scoping: (user_id, guild_id) composite key.

### 3. conversation_graph (threads + messages + summaries)
- Full conversation history in `conversation_threads`, `conversation_messages`,
  `conversation_summaries`.
- Supports tiered context assembly in `buildContext()` — lightweight tier
  gets last 3 messages, full tier gets 15 messages + 2 summaries + profile.
- Has FTS5 virtual table for full-text search.
- Per-guild-channel-user scoping: (user_id, guild_id, channel_id).

### 4. Realm NPC memory (realm_npc_memory)
- In-fiction NPC memory, scoped to the Realm game world. Never bleeds
  into the real-world persona context — this was an explicit separation.

### 5. Emotional context (user_emotional_context)
- Per-user per-guild emotional state tracking. Keyword + sentiment-based
  emotion detection producing `'neutral'`, `'happy'`, `'sad'`, `'anxious'`,
  `'angry'`, `'stressed'`. Generates tone directives for the system prompt
  via `getEmotionDirective()`.

### 6. Knowledge base (knowledge_base + knowledge_fts)
- Seeded Wikipedia-like topics + user-taught facts via `/learn`. Separate
  from per-user memory because it's server-independent factual knowledge.
  Has FTS5 for full-text search.

**Rule:** Do not let a future feature quietly merge two of these without
documenting why the wall came down. The separation was deliberate in each case.
```

```markdown
## Guardrails that are load-bearing, not decorative

- **Content-safety in role lines**: Realm roles explicitly state "fantasy
  violence only — no gratuitous gore, no romance or dating-style content."
  Confidant role line forbids romantic/sexual content. These are baked into
  `persona/roles.js`, not applied at a separate filter layer.
- **Hostile content detection** (`features/safety/hostileDetector.js`): 
  3-strike system silences abusive users for 1 hour. Checked in
  `mentionRouter.js` before any AI call. Exempts the apology flag from
  rate limits (flagForApology() still fires even after strikes).
- **Emotion detection** (`features/wisdom/emotionalIntelligence.js`):
  Generates AI tone directives but never gates responses — it guides tone,
  not permission. No sensitive-category exclusions in extraction are
  currently implemented (derived memory extraction may add this).
- **Reaction-only mode** (`features/authenticity/reactionController.js`):
  10% chance for casual intents to get a reaction instead of a full AI
  reply. Silent, no response.
- **Sleep mode**: Bot silences during configurable hours (SLEEP_START/
  SLEEP_END). Affects all automatic features — reactions, interjections,
  AI auto-respond. Slash commands explicitly blocked during sleep.
- **State decay** (`features/channelState/stateDecay.js`): Channel state
  transitions to Dormant only happen in the decay pass, never on message
  arrival. This was a bug fixed once — an arrival-time handler should
  never represent "this channel has been silent."
- **Attention gate** (`features/discordNative/attentionGate.js`): 
  Probability-based gate for AI channel auto-respond. Uses recency,
  message count escalation, channel activity decay, and sentiment boost.
  Reply-to-bot is a special early-exit signal (100% response). Does NOT
  bypass the user's opt-in preference (`proactive_opt_in = 1` required
  for proactive messages).
```

```markdown
## Known trade-offs, accepted deliberately

| Trade-off | Why accepted | What would change this |
|-----------|-------------|----------------------|
| user_memory ↔ memory_entries fragmentation | Etch writes to memory_entries only; 19 commands read from user_memory only. The two tables are out of sync and have been for some time | Real bug — needs a data migration to backfill user_memory from memory_entries, then redirect all reads to memory_entries, then drop user_memory |
| In-memory Maps for cooldowns (mention_cooldowns have SQLite table but reactionSystem.js uses in-memory Map) | Reactions are non-critical; losing cooldown state on restart is harmless | Move to SQLite for consistency |
| Conversation content stored in plaintext | Not a high-risk bot; encryption at rest would add complexity without clear threat model | If the bot processes PII or regulated data |
| No timezone-aware scheduling | All cron-like jobs (news digest, maintenance) run at fixed UTC times | Global user base would need per-user timezone resolution |
| Reaction cooldown is in-memory Map (reactionSystem.js) | Volatile state acceptable for reactions; lost on restart, recovers quickly | Move to SQLite cooldowns table |
| `roleTokenBudgets.consult` stayed at 400 despite context injection adding token overhead | Never increased after implementation; works because context is already in the prompt, the budget limits only the reply | Increase to 700-900 to give the model room |

## Cross-cutting bugs already found and fixed once

1. **Double-write races on in-memory-then-persisted state machines**: Always
   compute-once-write-once within a single function call. Found in the early
   channel state system where state was modified in memory and then persisted,
   leading to divergent state on restart.
2. **Concurrent-message double-processing**: Any handler that can receive two
   rapid inputs for the same user needs a per-user processing lock before
   doing anything stateful. Found in the mention router where rapid mentions
   could trigger concurrent AI calls for the same user.
3. **State computed on message arrival should never represent "this channel
   has been silent"**: Idle-only states (Dormant channel state) belong
   exclusively to a separate decay/cleanup job, never the arrival-time
   handler. Fixed in `features/channelState/stateDecay.js`.
4. **SQLite prepared statement spread vs .apply()**: `stmt.run.apply(db, vals)`
   does not work with better-sqlite3 — must use `stmt.run(...vals)` spread
   syntax. Found during code quality review of the attention system.
5. **Dead code in chatGate**: The old `chatGate` had signature
   `shouldRespond(content)` (string parameter), but `bot.js` called it with
   the full Discord message object. `content.includes('?')` always failed,
   caught by outer try/catch, effectively disabling the heuristic shortcut.
   Replaced by `attentionGate.js` in the attention system upgrade.
6. **user_memory ↔ memory_entries fragmentation (currently active bug)**: The
   migration from `user_memory` to `memory_entries` was partially done. `/etch`
   writes to `memory_entries`, but 19 commands still read from `user_memory`.
   A user who etches a fact won't see it in creative commands (joke, roast, etc.),
   while the AI DOES see it through `promptContext.js`. Fix requires:
   backfilling `user_memory` from `memory_entries`, retargeting all reads to
   `memory_entries`, then dropping `user_memory`.
```
