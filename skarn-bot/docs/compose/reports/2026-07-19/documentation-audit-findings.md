## 0. CRITICAL: Data Consistency Bug (Memory Fragmentation)

### 0.1 user_memory and memory_entries are out of sync

- `/etch` (`commands/etch.js` line 16 -> `features/etch/etch.handler.js` line 11) writes to **memory_entries** via `addMemoryEntry()` with `source='etch'`.
- `/forget` (`commands/forget.js` line 10 -> `features/forget/forget.handler.js` line 4) deletes from **memory_entries** via `deleteUserMemoryEntries()`.
- `promptContext.js` (lines 29-31) reads from **memory_entries** via `getMemoryEntries()` for AI context assembly.
- However, `database.js` still exports `getUserMemory()`, `addUserMemory()`, and `deleteUserMemory()` which operate on the **user_memory** table (schema lines 2-8). The `user_memory` table still exists in the schema and receives NO writes from any handler.
- **`addUserMemory()` is dead export** -- never called anywhere in the codebase.
- **`getUserMemory()` is also dead** -- never called from any command or feature file. `promptContext.js` uses `getMemoryEntries`, not `getUserMemory`.

- **Impact**: There is no data loss bug currently (the active path reads from `memory_entries`). But the `user_memory` table, its indexes, and its helper functions are dead code. If a future developer refactors and starts using `getUserMemory()` thinking it is the canonical memory store, they will get empty results while etch data sits in `memory_entries`.

- **This is not a documentation gap -- it is an active dead-code/data-fragmentation issue.**

## 2. CONTEXT.md Drift

### 2.1 "All state in SQLite" claim is not absolute

CONTEXT.md line 19: "All state in SQLite: No in-memory Maps, Sets, or JSON files may hold state that could be lost on restart."

The following in-memory state structures exist:
- `features/discordNative/reactionSystem.js` line 6: `const cooldowns = new Map();` -- per-channel reaction cooldowns
- `commands/search.js` line 13: `const cooldowns = new Map();` -- per-user search cooldown

These are ephemeral cooldowns (not durable state), so the statement is misleading rather than false. But a literal reading is contradicted.

### 2.2 buildContext() history references removed functions

CONTEXT line 32: "Merges the previous `collectContext()` (directive lines) and `assembleContext()` (conversation content) into one unified output."

No `collectContext()` or `assembleContext()` functions currently exist anywhere in the codebase. These were historical names that were indeed merged, but a developer reading CONTEXT.md cannot find these functions to verify the claim.

### 2.3 "memory_entries replaces user_memory + knowledge_graph" is partially inaccurate

CONTEXT line 11: "Replaces the previous `user_memory` + `knowledge_graph` separation."

The `user_memory` table (schema lines 2-8) still exists and is never written to. The `knowledge_graph` table (schema lines 234-245) still exists and IS still written to by `knowledgeGraph.js` and queried by `database.js` `getKnowledge()`. So `memory_entries` replaced `user_memory` only -- `knowledge_graph` is still active and separate.

### 2.4 Advice tier (socratic directive) is not populated

ADR-001 (0001-tiered-context-assembly.md) describes three tiers: Lightweight, Full, and Advice. The Advice tier says it includes a "socratic directive" for messages matching "should I", "what should" etc.

- CONTEXT.md doesn't mention the tiered system at all.
- `buildContext()` (`promptContext.js`) does NOT populate a `socraticLine` in its return value (lines 108-117).
- The `socraticLine` parameter is accepted by `buildSystemPrompt()` (identity.js line 58) but it is never populated by `buildContext()`.
- The Advice tier from the ADR does not exist in the current code.

### 2.5 Fast-path skippers list is incomplete

CONTEXT line 64 lists 4 fast-path skippers: `skarn opt in`, `skarn opt out`, `skarn chat mode`, `skarn status`.

Missing additional fast-path behaviors:
- DM handling (bot.js lines 251-273): routes to activation registry or mention handler before the normal flow
- `!` prefix messages (bot.js line 326): also routed through the activation registry

## 3. RPC_GUIDE.md Issues

### 3.1 RPC_GUIDE.md teaches creating a separate project

- RPC_GUIDE.md guides readers to create a new project from scratch (`mkdir discord-rpc`, `npm init -y`, `npm install discord-rpc`) rather than pointing readers to the existing `rich-presence.js` in the repo.
- The code sample (lines 24-42) differs from the actual `rich-presence.js`: missing `largeImageKey`, `largeImageText`, and `startTimestamp` fields.
- The guide says "Replace `YOUR_APPLICATION_ID`" but the actual `rich-presence.js` has the real Client ID hardcoded (line 3).

### 3.2 RPC_GUIDE.md is misleading as an internal reference

The README references RPC_GUIDE.md for "platform-agnostic setup" but the guide describes creating a separate project outside the repo. A reader trying to set up Rich Presence for the existing bot would be confused about whether to run the existing `rich-presence.js` or follow the tutorial to create a new one.

## 4. .env.example Drift

### 4.1 Missing critical variables

`.env.example` contains only: `DISCORD_TOKEN`, `CLIENT_ID`, sleep vars, `GOOGLE_CSE_KEY`, `GOOGLE_CSE_CX`.

Missing from example (all used in the actual code):
- `OPENAI_API_KEY` -- required for ALL AI functionality
- `AI_MODEL` -- used as default model everywhere (code defaults to `gpt-3.5-turbo`)  
- `AI_MODEL_COMPLEX` -- used in `modelRouter.js` for complex queries

### 4.2 Sleep mode defaults differ

`.env.example` sets `SLEEP_START=1`, `SLEEP_END=7` (sleep enabled, 1 AM - 7 AM).
Actual `.env` sets `SLEEP_START=0`, `SLEEP_END=0` (sleep disabled).
Code default (bot.js lines 64-65): `SLEEP_START=1`, `SLEEP_END=7` (matching .env.example).

This is intentional (different environments have different preferences), but worth noting that the example suggests a different default than the actual deployment.

## 5. Spec Document Issues

### 5.1 Web search spec describes DDG but code uses Google CSE

`docs/specs/2026-07-18-web-search.md` says:
- "Library: `duck-duck-scrape` npm package (unofficial DuckDuckGo wrapper, no API key needed)"
- "Why DuckDuckGo: Free, no API key, no rate limit concerns"

Actual code:
- `commands/search.js` and `features/search/searchEngine.js` use Google Custom Search Engine (CSE) with `GOOGLE_CSE_KEY` and `GOOGLE_CSE_CX`
- DuckDuckGo is used only as a FALLBACK (bot.js line 84: "will use DDG fallback") when Google CSE is not configured
- The `duck-duck-scrape` dependency is in `package.json` but serves as backup, not primary

### 5.2 Weather spec commands undocumented in README

The weather spec describes `/weather report` and `/weathertrack` commands. Actual command files exist (`commands/weather.js`, `commands/weathertrack.js`) and the code works, but these commands are:
- Not listed in README.md command tables
- Not mentioned in any user-facing documentation

### 5.3 Friend Tiers spec shows no implementation evidence

`docs/specs/2026-07-19-friend-tiers-permissions-spec-sonnet-5-medium.md` (450 lines) describes a 5-tier system (Stranger -> Acquaintance -> Regular -> Trusted -> Warband) with automatic promotion, tier-based rate limits, and admin permission flags.

There is NO evidence of this system in the codebase:
- No tier-related tables in `skarn-schema.sql`
- No tier-related code in `database.js`
- No tier logic in any feature file
- The `friends` table (schema lines 450-455) is a simple flat list with code/name/power/note -- no tier concept
- This spec appears to be a design document for a feature that has NOT been implemented

### 5.4 Derived Memory spec shows partial implementation

`docs/specs/2026-07-19-derived-memory-spec-sonnet-5-medium.md` (532 lines) describes a passive knowledge extraction system with relevance-ranked retrieval and embedding-based similarity.

Partial implementation evidence:
- `features/memory/memoryExtractor.js` exists (10 lines) -- delegates to `knowledgeGraph.extractAndStore()`
- `features/intelligence/knowledgeGraph.js` exists -- extracts structured entities from conversations
- The `memory_entries` table has the schema described in the spec
- However, the embedding-based similarity ranking and confidence-scoring described in the spec are NOT implemented in the code found

### 5.5 Intelligence/Authenticity spec has unimplemented items

`docs/specs/2026-07-18-intelligence-authenticity-design.md` describes:
- `features/intelligence/intentClassifier.js` -- **does not exist** (listed as new file in spec)
- `features/social/serverAwareness.js` -- **does not exist** (listed as new file in spec)  
- `intent_cache` table -- **does not exist** in schema (listed as new table in spec)

### 5.6 Command Activation spec exists in compose directory

`docs/compose/specs/2026-07-19-command-activation-gate-audit-spec.md` -- spec exists but its implementation status is unclear from code scan alone.

## 8. Terminology Inconsistencies

### 8.1 "Dormant" state in CONTEXT.md

CONTEXT.md line 51 says: "Dormant is only ever set by the decay pass, never by incoming messages."

This is correct per `channelState/stateDecay.js` and `bot.js` -- however, the term "Dormant" is not explained anywhere in user-facing documentation.

### 8.2 "Weathering" state defined but not in CONTEXT.md

CONTEXT.md line 23 mentions "Weathering state detection" and the channel state machine has 4 states (Dormant, Attentive, Charged, Weathering). But line 51 only discusses Dormant and Attentive/Charged -- "Weathering" is not explicitly described in terms of state transitions.

### 8.3 Familiarity cap description

CONTEXT.md line 50: "Capped at +3/day for message-based gains. Decays 1/day of inactivity."

The familiarity mechanic has +1 for etch, +2 for mention, +1 for AI command (per the spec doc `2026-07-18-persona-depth.md`). But CONTEXT.md does not detail these per-action values -- only the message cap and decay.
